# 嵌入式开发学习笔记

## 一、环境部署

本次环境部署尝试了两种方案：**Win11 + WSL2 (Linux)** 和 **Windows原生环境**。

### 3月23日

#### Linux (WSL2) 环境

- `eim install` 指令报错，库拉取失败，`tool.json` 无法读取
- 初步判断是代理问题
- **困境**：主机开启代理后，WSL2 中 Git 证书验证不通过；不开启代理，又无法连接 GitHub 服务器
- WSL2 代理配置确认无误（主机IP:代理端口）
- 尝试拷贝并配置代理证书 —— **失败**，证书仍无法通过（原因不明）
- 考虑改用 SSH 代替 HTTPS，配置 SSH Key 后拉取速度过慢，暂时放弃

#### Windows 环境

- 遇到 Python 版本不匹配问题

---

### 3月24日

#### Linux (WSL2) 环境

- 转向**离线下载**方案
- Git 绕过证书验证成功 clone 仓库
- 离线执行 `install.sh` 脚本时，Python 又遇到证书问题，`dist` 与 `tool` 从 GitHub 拉取失败
- 尝试为 Python 配置证书 —— **仍旧失败** (TAT)
- **最终方案**：在 Windows 主机手动下载所有依赖，再 `cp` 到 WSL2 中
- 依次执行 `install.sh` 和 `export.sh` 脚本
- **Success!**
- 配置 VS Code 连接 WSL2，安装 IDF 插件，复制 `examples/get-started/hello-world` 示例，成功运行 `idf.py build`

#### Windows 环境

- 更换 Python 版本
- 安装进度卡在 85%
- 参考 issues #613 & #596：[idf-im-ui #596](https://github.com/espressif/idf-im-ui/issues/596#issuecomment-4106193762)
- 更换版本后成功安装
- 完成 VS Code 插件安装与环境配置

#### 阶段性结论

> Windows 环境下 IDF 部署确实更为便捷，但考虑到开发环境与编译烧录效率，**更推荐使用 Linux**。
>
> WSL2 的网络环境较为复杂，虽然将代理修改为主机IP:端口，仍存在诸多问题（比如至今未解决的证书问题 TAT）。

## 二、调试与 Debug 心得

**一个字：累，但值得。**

作为一名**零嵌入式经验**的开发者，此次招新实验 + 初级项目让我收获颇丰。

### 嵌入式 vs 软件工程

两者既有相似之处，又存在本质差异：

| 维度 | 相似之处 | 不同之处 |
|------|----------|----------|
| 逻辑思维 | 面对需求与问题设计程序逻辑 | 情境更加具体，抽象方法论与开发经验深度契合 |
| 学习过程 | 调用工具 API 时需要阅读文档 | 必须了解开发板核心特性、架构、基本原理 |
| 技术栈 | 编程语言基础相通 | 需学习板上通信、串口通信等诸多协议 |
| 资源管理 | 基本的内存概念 | **极度注重**内存分配与缓冲区管理，对数据存储与传输有极高要求 |

**举个栗子**：I2S 通信中数据位宽与位深的概念，不同格式下内存中每一位的 0 与 1 分别代表左右声道 —— 这种细节在纯软件工程中很少涉及。

### 关于开发方式

目前还无法做到短时间内独立实现大型项目从 0 到 1 的构建。现阶段借助 Claude Code 辅助代码构建、研习、调试，直至程序成功运行。

## 三、学习难点总结

1. **丰富且抽象的通信协议**
   - 想要熟练开发，不能只会调用 API
   - 必须深入了解协议特点、工作原理，才能更好地运用
   - 例如：具体通信频率的配置

2. **大量繁多的结构体配置**
   - 引脚的 `init` 与 `config`，各种功能的 `init` 与 `config`
   - 对于初学者而言可谓望而生畏
   - 阅读文档或跳转到定义了解功能实现，过程枯燥且头大

3. **数据形式与内存分配的和谐**
   - 对硬件开发与内核操作等极端环境至关重要
   - 零经验者难以掌握与理解

4. **OS 核心概念**
   - 时钟、中断、寄存器等操作系统内核概念，确实难啃
